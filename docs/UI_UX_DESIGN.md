# Rate My Unhinged Decision -- Complete UI/UX Design Document

## DESIGN PRINCIPLE #0: NO AI SLOP

This app must NOT look like every other AI-generated website. Specific anti-slop rules:

**DO NOT USE:**
- Gradient text on headings (the "AI startup" cliche)
- Glassmorphism cards with blurred backgrounds everywhere
- Floating orbs, mesh gradients, or aurora backgrounds
- Generic hero sections with "Powered by AI" badges
- Smooth rounded everything with no edges or tension
- Pastel-on-white "clean minimal" aesthetic
- Decorative SVG blobs or waves
- "Built with love" or sparkle emojis in the footer
- Animated gradient borders on cards
- The purple-blue-pink gradient combo (every AI tool uses this)
- Cookie-cutter component library aesthetics (shadcn defaults without customization)
- Overly polished, sterile interfaces that feel like a template

**DO USE:**
- Raw, opinionated typography choices (monospace body text IS the personality)
- Intentional roughness: uppercase labels with wide tracking feel like a CASE FILE, not a SaaS dashboard
- Hard color choices (red, yellow, purple -- not safe blues and grays)
- The monospace + geometric sans combo reads as "forensic analysis tool" not "another AI wrapper"
- Brutalist-inspired spacing: tight headings (1.0 line-height), generous body text (1.625)
- Functional animations only: score counter, typewriter, shimmer bar -- no decorative floating particles
- Left-aligned text (not centered everything like a landing page template)
- One-column 480px layout at all sizes (apps have focus, templates have sections)
- Personality in microcopy: "JUDGE ME", "SLOW DOWN, CHAOS AGENT", "consulting historical records..."
- The dashed-border recommendation box and solid-border verdict card are DIFFERENT styles intentionally -- visual variety through function, not decoration

**The litmus test:** If you removed the app name and showed someone the UI, could they tell it apart from 100 other AI apps? If no, redesign. This app should look like it was designed by someone with opinions, not generated by a prompt.

## SECTION 1: Design System Definition

### 1.1 Color Palette

**The Aesthetic: "Crime Scene Report Meets Late-Night Talk Show"**

This app judges unhinged decisions. It should feel like a darkened interrogation room with a spotlight -- dramatic, slightly dangerous, but ultimately entertaining. The palette draws from courtroom exhibits, police procedural title cards, and the neon glow of bad decisions made at 2 AM.

**Why dark mode only (detailed justification in Section 7):** Unhinged decisions happen at night. The dark background creates theater -- a stage for the verdict. It also makes the score's color gradient pop dramatically, which is impossible against a white background.

```
Design Tokens (CSS Custom Properties)
--------------------------------------

/* Core Brand */
--color-primary:         #FF3B3B    /* Verdict red -- hot, urgent, alarming. The "you're cooked" color. */
--color-secondary:       #FFB800    /* Caution yellow -- warning tape, hazard signs, "proceed at your own risk" */
--color-tertiary:        #7B61FF    /* Unhinged purple -- chaos energy, the irrational, wildcard accent */

/* Backgrounds */
--color-bg-base:         #0D0D0F    /* Near-black with blue undertone, not pure #000 (reduces eye strain per WCAG best practices) */
--color-bg-surface:      #18181B    /* Elevated surface -- cards, modals */
--color-bg-elevated:     #27272A    /* Higher elevation -- input fields, hover states */
--color-bg-glass:        rgba(24, 24, 27, 0.85)  /* Glassmorphism sparingly -- only on shareable card overlay */

/* Text */
--color-text-primary:    #FAFAFA    /* Off-white, not pure #FFF (contrast: 18.5:1 against bg-base, exceeds WCAG AAA) */
--color-text-secondary:  #A1A1AA    /* zinc-400 equivalent -- subtext, labels */

/* Semantic */

--color-warning:         #FFB800    /* Same as secondary -- reuse for consistency */

--color-score-100:       #FF3B3B    /* Red -- "historically unhinged, congratulations" */
```

**Rationale:**
- **Red primary (#FF3B3B):** Not the generic startup red. This is slightly more orange-warm than a cold #EF4444 -- it reads as "alarm" not "error." It references warning labels, content warnings, and the red stamp on a classified document. Research: warm reds increase arousal and attention (Elliot & Maier, 2014, "Color and Psychological Functioning").
- **Yellow secondary (#FFB800):** Caution tape. Hazard signs. The literal visual language of "this might be a bad idea." Works beautifully against dark backgrounds with contrast ratio 10.2:1.
- **Purple tertiary (#7B61FF):** Used sparingly -- only for the comparison figure percentage bars and the "unhinged" descriptor text. Purple is culturally coded as "irrational, mystical, chaotic" (Heller, 2009, "Psychology of Color") -- perfect for this app's tone.
- **Dark background (#0D0D0F):** Not pure black. Google's Material Design research shows pure black backgrounds create excessive contrast that strains eyes during extended reading. The slight blue undertone adds depth and prevents the "OLED void" effect.

### 1.2 Typography

**Heading Font: Space Grotesk (Google Fonts)**
- Why: Geometric sans-serif with just enough quirk. The slightly squared terminals and distinctive 'g' and 'a' give it personality without sacrificing readability. It feels technical and authoritative -- like a forensic report header -- but modern. Available from Google Fonts with `next/font/google` for zero layout shift.
- Weights used: 700 (headings, score display), 500 (subheadings, labels)

**Body Font: JetBrains Mono (Google Fonts)**
- Why: Monospaced body text is an intentional aesthetic choice. It reads like a terminal output or police report -- fitting for an app that "analyzes" your decisions. Monospace fonts also enforce consistent character widths, making the score counter animation feel more mechanical and satisfying. The "verdict" text in monospace feels like it was typed by an algorithm, reinforcing the "AI analyzed you" conceit.
- Weights used: 400 (body text, descriptions), 700 (emphasized text, verdict)
- Caveat: Monospace for body text is an intentional F-pattern violation -- users read monospace ~10% slower (Perea et al., 2012). This is acceptable because (a) the text is short (verdict + profile are 2-3 sentences each), and (b) the slower reading actually benefits engagement in this context -- users savoring their roast.

```

----------

--text-sm:    0.875rem  / 14px   (line-height: 1.25rem)   -- labels, secondary info
--text-base:  1rem      / 16px   (line-height: 1.625rem)  -- body text (26px line-height for monospace readability)
--text-lg:    1.125rem  / 18px   (line-height: 1.75rem)   -- emphasized body
--text-xl:    1.25rem   / 20px   (line-height: 1.75rem)   -- subheadings
--text-2xl:   1.5rem    / 24px   (line-height: 2rem)      -- section titles

--text-6xl:   6rem      / 96px   (line-height: 1)         -- score number (desktop)
```

**Weight Rules:**
- 700 only: Page title, score number, verdict one-liner, CTA buttons, section labels
- 500: Subheadings, comparison figure names, percentage labels
- 400: Everything else (body, descriptions, form text, meta)
- Never use 300/200 -- too thin for dark backgrounds (contrast loss per WCAG)

**Letter Spacing:**
- Headings (Space Grotesk): `-0.02em` (tighter -- feels more impactful)
- Body (JetBrains Mono): `0` (monospace has built-in spacing, don't fight it)
- Labels/All-caps text: `0.1em` (readability boost for uppercase, per NN Group typography guidelines)

**Line Height Rules:**
- Monospace body text: 1.625 (generous -- monospace needs breathing room)
- Headings: 1.0-1.1 (tight -- dramatic, poster-like)
- Multi-line descriptions: 1.5 (balanced readability)

### 1.3 Spacing & Layout

**Base Unit: 4px grid**
All spacing values are multiples of 4px. This creates visual rhythm without being overly constrained.

```
Spacing Scale (Tailwind classes)
---------------------------------
0.5 = 2px    (micro-gaps within tight elements)
1   = 4px    (icon-to-label gap)
2   = 8px    (tight internal padding)
3   = 12px   (default internal padding)
4   = 16px   (card internal padding mobile, element spacing)
5   = 20px   (section element spacing)
6   = 24px   (card internal padding desktop)
8   = 32px   (section spacing mobile)
10  = 40px   (section spacing transition)
12  = 48px   (section spacing desktop)
16  = 64px   (major section breaks)
20  = 80px   (hero top padding desktop)
24  = 96px   (page top padding desktop)
```

**Container:**
```
--container-max:     480px    (mobile-optimized single column -- this is NOT a dashboard, it's a focused input-output tool)
--container-padding: 16px     (mobile), 24px (desktop)
```

Rationale for 480px max: Research on optimal line length for readability is 50-75 characters per line (Baymard Institute, 2024). At 16px monospace, 480px container yields approximately 45-50 characters per line -- slightly under optimal, which is correct because the content is short-form (verdict, profile). Wider containers would create awkward short paragraphs with too much whitespace.

**Mobile vs Desktop Padding:**
- Page padding: `px-4` (16px) mobile, `px-6` (24px) at 768px+
- Card padding: `p-4` (16px) mobile, `p-6` (24px) at 768px+
- Section gaps: `gap-8` (32px) mobile, `gap-12` (48px) at 768px+

**Card Styling:**
- Border radius: `rounded-2xl` (16px) -- generous, modern, but not comically round
- Border: `1px solid rgba(255, 255, 255, 0.08)` -- subtle edge definition against dark bg
- Background: `--color-bg-surface` (#18181B)

## SECTION 2: User Journey Mapping

### 2.1 Entry Point

**Primary traffic sources (in priority order):**
1. **Shared card on Instagram Stories** -- someone sees a friend's verdict card, taps the "ratemyunhinged.app" watermark or swipe-up link
2. **Shared link on Twitter/X** -- link preview shows OG image with a provocative score
3. **Direct link from NodeSparks** -- curious visitors from the challenge page

**First impression (above-the-fold, no scroll required):**

The user lands and sees, in this order (respecting F-pattern reading):
1. **App title** (top-left aligned): "RATE MY UNHINGED DECISION" in Space Grotesk 700, uppercase, tracking wide
2. **Tagline** (directly below, left-aligned): "confess. get judged. share the damage." in JetBrains Mono 400, lowercase, text-secondary color
3. **The textarea** (immediately below -- no scroll on any phone 375px+): Large, prominent, with a compelling placeholder
4. **Submit button** (full-width below textarea): "JUDGE ME" in Space Grotesk 700

**What makes them try it:**
- The copy tone is irreverent and inviting -- lowercase tagline signals "this is not corporate"
- The placeholder text teases an example: `"I proposed to my ex at their wedding to someone else..."`
- Zero friction -- no signup, no account, no cookie consent banner. Type and go.
- If they arrived via a shared card, they already have FOMO ("I want my own score")

**Why no social proof section:** This is a weekend micro-app, not a SaaS product. Social proof ("10,000 decisions rated!") adds clutter and requires real data. The shared cards ARE the social proof -- they exist on the user's Instagram feed. Adding a counter would feel like startup theater.

### 2.3 Processing Phase

**Duration target: 2-4 seconds** (Groq is fast, but we add artificial delay to build anticipation)

**Minimum display time: 2 seconds.** Even if the API responds in 800ms, hold the loading state for at least 2 seconds. Research: Jakob Nielsen's response time thresholds (1993, updated 2014) show that 0.1-1 second feels "instant," but for this app, instant feels unearned. The wait creates anticipation and makes the result feel more considered, more legitimate -- like the AI actually thought about it.

**Loading animation design:**

### 2.5 Sharing Phase

**Mobile sharing (primary flow -- Web Share API):**
1. User taps "SHARE YOUR VERDICT" button
2. Button text changes to "GENERATING..." with a spinner (200ms)
3. `html-to-image` renders the hidden 1080x1920 ResultCard component to PNG blob
4. `navigator.share({ files: [blob], title: "Rate My Unhinged Decision", text: "I scored {score}/100 on the unhinged scale. How unhinged are you?" })`
5. Native share sheet appears (Instagram Stories, iMessage, WhatsApp, etc.)
6. On success: button briefly shows "SHARED" with a checkmark, then reverts
7. On cancel: button reverts silently (no error state for user-initiated cancel)

**Desktop sharing (fallback -- clipboard):**
1. User clicks "SHARE YOUR VERDICT"
2. Card is generated as PNG, auto-downloaded to their device
3. Share URL is copied to clipboard
4. Button text changes to "CARD DOWNLOADED + LINK COPIED" for 3 seconds
5. Toast-style notification at bottom of viewport confirms the action

**Download option:**
- "download card" ghost button below share button
- Directly downloads the 1080x1920 PNG with filename `unhinged-score-{score}.png`
- No intermediate steps

**Hook text on the card (what makes viewers try the app):**
Bottom of the card includes: `"How unhinged are you? -> ratemyunhinged.app"` in text-sm. This is NOT a "powered by" -- it's a dare. The question form creates curiosity gap (Loewenstein, 1994, "The Psychology of Curiosity"). Viewers who see a friend's 87/100 score immediately want to know their own number.

## SECTION 4: Shareable Card Design (CRITICAL)

### 4.1 Dimensions and Rendering

- **Render size:** 1080 x 1920 pixels (9:16, Instagram Story native resolution)
- **Rendering method:** `html-to-image` library (`toPng()`) targeting a hidden `<div>` with `position: fixed; left: -9999px; width: 1080px; height: 1920px;`

- **File size target:** Under 1MB (Instagram Stories recompresses anything larger)

### 5.5 Score Counter Animation

```typescript
// Score counts from 0 to final value using requestAnimationFrame
// Easing: easeOutExpo -- fast start, dramatic slowdown at end
// Duration: 800ms

function easeOutExpo(t: number): number {
  return t === 1 ? 1 : 1 - Math.pow(2, -10 * t);
}

function animateScore(target: number, onUpdate: (value: number) => void) {
  const duration = 800;
  const start = performance.now();

  function tick(now: number) {
    const elapsed = now - start;
    const progress = Math.min(elapsed / duration, 1);
    const easedProgress = easeOutExpo(progress);
    const current = Math.round(easedProgress * target);
    
    onUpdate(current);

    if (progress < 1) {
      requestAnimationFrame(tick);
    }
  }

  requestAnimationFrame(tick);
}
```

## SECTION 7: Dark Mode Decision

**Decision: Dark mode only. No light mode. No toggle.**

**Justification:**

1. **Thematic alignment.** This app is about confessing unhinged decisions. The dark background creates the feeling of a late-night confession booth, a dimly lit interrogation room, a Reddit thread at 3 AM. Light mode would feel like a job application form. The mood IS the product.

2. **Score gradient visibility.** The score color gradient (green -> yellow -> orange -> red) pops dramatically against a dark background. Against white, these same colors would look washed out and require much darker/more saturated values to maintain contrast, losing their neon "danger sign" quality.

3. **Shareable card consistency.** The card is designed as a dark graphic. If the app had a light mode but the card was dark, there would be a visual disconnect. If the card were also light, it would disappear in Instagram Stories (which have predominantly light content). Dark cards stand out in feeds (Section 4.6 reasoning).

4. **OLED advantage.** 72% of smartphones sold in 2024 have OLED screens (Counterpoint Research). Dark UI on OLED uses less power and looks richer (true blacks). The app's target demographic (young, mobile-first, Instagram users) disproportionately has OLED phones.

5. **Reduced complexity.** A light mode doubles the design surface area and introduces potential contrast failures. For a weekend build, one mode done excellently beats two modes done adequately.

**Dark mode implementation details:**

```css
:root {
  color-scheme: dark;
}

/* Force dark regardless of system preference */
html {
  background-color: #0D0D0F;
  color: #FAFAFA;
}

/* Prevent browser auto-darkening */
meta[name="color-scheme"] content="dark"
/* Also add to <head>: <meta name="theme-color" content="#0D0D0F"> */
```

### Score Color Interpolation Utility

```typescript
// lib/utils.ts

export function getScoreColor(score: number): string {
  const stops = [
    { at: 0,   color: [34, 197, 94] },   // #22C55E
    { at: 25,  color: [132, 204, 22] },   // #84CC16
    { at: 50,  color: [255, 184, 0] },    // #FFB800
    { at: 75,  color: [249, 115, 22] },   // #F97316
    { at: 100, color: [255, 59, 59] },    // #FF3B3B
  ];

  // Find the two stops the score falls between
  let lower = stops[0];
  let upper = stops[stops.length - 1];

  for (let i = 0; i < stops.length - 1; i++) {
    if (score >= stops[i].at && score <= stops[i + 1].at) {
      lower = stops[i];
      upper = stops[i + 1];
      break;
    }
  }

  const range = upper.at - lower.at;
  const progress = range === 0 ? 0 : (score - lower.at) / range;

  const r = Math.round(lower.color[0] + (upper.color[0] - lower.color[0]) * progress);
  const g = Math.round(lower.color[1] + (upper.color[1] - lower.color[1]) * progress);
  const b = Math.round(lower.color[2] + (upper.color[2] - lower.color[2]) * progress);

  return `rgb(${r}, ${g}, ${b})`;
}

export function getScoreGlow(score: number): string {
  const color = getScoreColor(score);
  // Extract RGB values and create glow with 20% opacity
  return `0 0 60px ${color.replace('rgb', 'rgba').replace(')', ', 0.2)')}`;
}
```
